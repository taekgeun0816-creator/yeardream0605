# 제약 조건 비교 데모 — 회원가입 / 로그인

Day 05 **데이터베이스 구성하기(제약 조건)** 강의를 보조하는 Streamlit 미니 프로젝트입니다.  
동일한 회원가입·로그인 UI에서 **good.db**(적절한 제약)와 **bad.db**(과도한 제약)의 결과 차이를 직접 비교해볼 수 있습니다.

## 프로젝트 구조 및 각 파일 역할

```
constraint_demo/
├── app.py
├── database/
│   ├── good_db.py
│   ├── bad_db.py
│   ├── good.db
│   └── bad.db
├── pyproject.toml
├── uv.lock
└── README.md
```

- **app.py**
  - Streamlit 기반의 메인 프론트엔드 파일
  - 두 DB(good/bad)를 동시에 초기화하여 UI에서 직접 비교 가능하게 함
  - 회원가입·로그인·회원 목록 조회 기능을 제공
- **database/good_db.py**
  - `good.db`와의 상호작용 담당(초기화 포함)
  - 회원 테이블에 **적절한 제약조건**(ex: 최소 길이, 일반 이메일 형식, 필수 입력 등) 적용
  - CRUD 연산 함수 제공
- **database/bad_db.py**
  - `bad.db`와의 상호작용 담당(초기화 포함)
  - 회원 테이블에 **과도하거나 잘못된 제약조건**(ex: 과하게 긴 아이디, 이메일 도메인 강제, plan_id FK 오류 등) 적용
  - CRUD 연산 함수 제공
- **database/good.db**
  - `good_db.py` 실행시 자동 생성되는 SQLite DB 파일
  - 적절한 제약(실무 맞춤형) 구조 확인용
- **database/bad.db**
  - `bad_db.py` 실행시 자동 생성되는 SQLite DB 파일
  - 과도/잘못된 제약 구조 확인용
- **pyproject.toml**
  - 의존성 및 프로젝트 메타데이터 관리
  - Streamlit 및 uv 등 주요 패키지 명시
- **uv.lock**
  - 의존성 고정 파일(uv 패키지 매니저용)
- **README.md**
  - 프로젝트 안내 및 실습 안내 문서

## 설치 (uv)

```bash
cd constraint_demo
uv sync
```

## 실행

```bash
uv run streamlit run app.py
```

브라우저에서 `http://localhost:8501` 접속

## 개발 시 주의 (Streamlit warning 방지)

Streamlit 1.40+ / 1.58 기준으로 아래 API를 사용합니다. deprecated 인자는 **사용하지 않습니다**.

| 사용 금지 (deprecated) | 대신 사용 |
|------------------------|-----------|
| `use_container_width=True` | `width="stretch"` |
| `use_container_width=False` | `width="content"` |
| `use_column_width="auto"` 등 | `width="stretch"` 또는 `width="content"` |
| `st.experimental_*` | `st.*` 공식 API |

현재 코드 적용 예:

```python
st.dataframe(users, width="stretch", hide_index=True)
```

새 UI 요소(차트, 버튼, 이미지 등) 추가 시에도 [Streamlit API Reference](https://docs.streamlit.io/develop/api-reference)의 `width` / `height` 파라미터를 우선 확인하세요.

## DB별 제약조건 비교 요약

| 구분       | good.db          | bad.db                                     |
|------------|------------------|--------------------------------------------|
| 아이디     | 3자 이상         | 10~12자, 소문자·숫자만 허용               |
| 이메일     | 일반 형식        | `@school.ac.kr` 도메인만 허용              |
| 비밀번호   | 4자 이상         | 정확히 8자, 숫자+대문자 필수               |
| FK         | 없음             | `plan_id` DEFAULT `'VIP'` (존재하지 않아 항상 실패) |

## 테스트 시나리오

아래 시나리오는 **같은 화면·같은 입력**으로 Good DB와 Bad DB의 결과 차이를 확인하기 위한 실습 순서입니다.  
각 단계마다 **실행된 SQL**과 **오류/성공 메시지**를 함께 읽어 보세요.

### 사전 준비

1. `uv run streamlit run app.py` 실행 후 브라우저에서 `http://localhost:8501` 접속
2. 화면 좌측 **Good DB**, 우측 **Bad DB** 패널이 나란히 보이는지 확인
3. 이전 실습 데이터가 남아 있다면 하단 **「두 DB 모두 초기화」** 버튼으로 초기화

---

### 시나리오 1 — 정상 회원가입 (Good DB 성공)

| 항목 | 입력값 |
|------|--------|
| 아이디 | `kim` |
| 이메일 | `kim@test.com` |
| 비밀번호 | `1234` |
| 비밀번호 확인 | `1234` |

**Good DB → 회원가입 탭 → 가입하기**

| 확인 포인트 | 기대 결과 |
|-------------|-----------|
| 결과 | `회원가입 성공` |
| SQL | `INSERT INTO users (username, email, password) VALUES (?, ?, ?)` |
| 적용 제약 | NOT NULL · UNIQUE · CHECK(3자 이상 / 4자 이상 / 이메일 형식) · PK(AUTOINCREMENT) |

**회원 목록** 탭에서 `kim` / `kim@test.com` 이 등록됐는지 확인합니다.

---

### 시나리오 2 — 동일 입력, Bad DB 실패 (CHECK 제약)

시나리오 1과 **완전히 같은 값**으로 **Bad DB**에 가입을 시도합니다.

| 확인 포인트 | 기대 결과 |
|-------------|-----------|
| 결과 | CHECK 제약 위반 오류 (가입 실패) |
| 아이디 | `kim`(3자) → 10~12자 미만 |
| 이메일 | `kim@test.com` → `@school.ac.kr` 아님 |
| 비밀번호 | `1234`(4자) → 정확히 8자 + 대문자·숫자 조건 불충족 |

**학습 포인트:** UI는 동일해도 DB 설계(제약 조건)에 따라 결과가 달라집니다.

---

### 시나리오 3 — Bad DB 조건을 맞춰도 FK로 실패

Bad DB의 CHECK를 모두 만족하는 값으로 다시 시도합니다.

| 항목 | 입력값 |
|------|--------|
| 아이디 | `kimminjun01` (10자, 소문자·숫자) |
| 이메일 | `minjun@school.ac.kr` |
| 비밀번호 | `Pass1234` (8자, 대문자·숫자 포함) |

**Bad DB → 가입하기**

| 확인 포인트 | 기대 결과 |
|-------------|-----------|
| CHECK | 통과 |
| FK | **실패** — `plan_id` 기본값 `'VIP'`가 `membership_plans`에 없음 |
| 오류 메시지 | FOREIGN KEY 위반 관련 안내 |

**학습 포인트:** CHECK를 통과해도 **잘못된 FK + DEFAULT 설계** 때문에 INSERT 자체가 불가능합니다.  
앱은 `plan_id`를 보내지 않는데, DB가 존재하지 않는 값을 강제하는 것이 문제입니다.

---

### 시나리오 4 — UNIQUE 제약 (중복 가입)

**Good DB**에서 시나리오 1과 같은 `kim` / `kim@test.com` / `1234` 로 **한 번 더** 가입합니다.

| 확인 포인트 | 기대 결과 |
|-------------|-----------|
| 결과 | UNIQUE 제약 위반 |
| 메시지 | `이미 사용 중인 아이디입니다.` 또는 `이미 등록된 이메일입니다.` |

아이디만 바꾸고(`kim2`) 이메일을 같게 넣으면 **이메일 UNIQUE** 오류를,  
이메일만 바꾸고 아이디를 같게 넣으면 **아이디 UNIQUE** 오류를 각각 확인할 수 있습니다.

---

### 시나리오 5 — NOT NULL (빈 값)

**Good DB → 회원가입**에서 아이디·이메일·비밀번호 중 하나를 비운 채 **가입하기**를 누릅니다.

| 확인 포인트 | 기대 결과 |
|-------------|-----------|
| 결과 | `모든 항목을 입력하세요.` (앱 단 검증) |
| DB INSERT | 실행되지 않음 |

비밀번호와 비밀번호 확인이 다를 때는 `비밀번호가 일치하지 않습니다.` 가 먼저 표시됩니다.

---

### 시나리오 6 — 로그인

**Good DB**에서 시나리오 1 가입 후 **로그인** 탭으로 이동합니다.

| 테스트 | 입력 | 기대 결과 |
|--------|------|-----------|
| 성공 | 아이디 `kim` / 비밀번호 `1234` | 로그인 성공 + 사용자 JSON |
| 성공(이메일) | `kim@test.com` / `1234` | 동일하게 성공 |
| 실패 | `kim` / `wrong` | `아이디 또는 비밀번호가 올바르지 않습니다.` |

Bad DB는 시나리오 3까지 가입이 되지 않으므로, 로그인 성공 케이스는 **Good DB 위주**로 확인합니다.

---

### 시나리오 7 — 초기화 후 재실습

1. 하단 **「두 DB 모두 초기화 (회원 데이터 삭제)」** 클릭
2. **회원 목록** 탭에서 Good/Bad 모두 비어 있는지 확인
3. 시나리오 1부터 다시 진행

---
